## 1. IF-001 행정포털 SSO

### 1.1 개요

| 항목 | 내용 |
|------|------|
| 외부 시스템 | 담양군 행정포털 |
| 연동 목적 | 내부 직원(공무원) SSO 인증 |
| 방식 | OAuth2 / SAML / 자체 표준 (협의) |
| 주기 | 사용자 로그인 시점 (이벤트 기반) |
| 관련 SFR | SFR-015 |
| 본 시스템 화면 | AUTH01-V SSO 로그인 처리 |

### 1.2 인증 흐름 (OAuth2 기준 — 협의 후 확정)

```
1. 사용자 → 행정포털 메뉴에서 "담양 안전관리시스템" 클릭
2. 행정포털 → 본 시스템 /sso/callback?code={auth_code} 리다이렉트
3. 본 시스템 → 행정포털 토큰 발급 API 호출
   POST {portal}/oauth2/token
   Body: { code, client_id, client_secret, redirect_uri, grant_type='authorization_code' }
4. 행정포털 → access_token, refresh_token 응답
5. 본 시스템 → 사용자 정보 조회 API 호출
   GET {portal}/userinfo
   Header: Authorization: Bearer {access_token}
6. 행정포털 → 사용자 정보 응답 (1.3 페이로드 참조)
7. 본 시스템 → users 테이블 UPSERT (auth_source='SSO')
8. 본 시스템 → 세션 발급 + 권한별 대시보드 진입
```

### 1.3 사용자 정보 응답 페이로드 (예상 — 협의 후 확정)

```json
{
  "external_id": "hong.gd",           // 행정포털 사용자 ID
  "name": "홍길동",
  "email": "hong@damyang.go.kr",
  "phone": "063-xxx-xxxx",
  "department": {
    "external_id": "DEPT-0123",       // 행정포털 부서 ID
    "name": "자치행정과",
    "parent_external_id": "DEPT-0100"
  },
  "position": "과장",                  // 직급
  "position_code": "5급",
  "duty": "안전보건담당",              // 직무 (SHM 매핑용)
  "is_active": true
}
```

### 1.4 권한 자동 매핑

```
permission_rules 마스터 조회:
  WHERE portal_position = userinfo.position
  ORDER BY priority DESC LIMIT 1

매핑 결과를 users.permission_role 설정
이후 변경은 CEO/GM이 ORG02-D에서 수동 (SSO 동기화로 덮어쓰지 않음)
```

### 1.5 오류 처리

| 상황 | 처리 |
|------|------|
| 토큰 만료 | 행정포털 로그인 페이지 재리다이렉트 |
| 토큰 검증 실패 | AUTH99-V `?code=sso_token_invalid` |
| 사용자 정보 조회 실패 | AUTH99-V `?code=sso_api_error` + 재시도 버튼 |
| 비활성 사용자 (`is_active=false`) | AUTH99-V `?code=user_inactive` |

### 1.6 협의 필요 사항

- [ ] 인증 프로토콜 확정 (OAuth2 / SAML / 자체)
- [ ] API 엔드포인트 URL
- [ ] client_id / client_secret 발급
- [ ] 사용자 정보 페이로드 스펙
- [ ] 세션 길이 정책 (refresh_token 만료)

## 2. IF-002 행정포털 조직도 동기화

### 2.1 개요

| 항목 | 내용 |
|------|------|
| 외부 시스템 | 담양군 행정포털 조직도 |
| 연동 목적 | departments / users 마스터 자동 동기화 |
| 방식 | REST API (Pull) |
| 주기 | 매일 06:00 + 수동 동기화 |
| 관련 SFR | SFR-006 |
| 본 시스템 화면 | ORG04-S 동기화 설정 |

### 2.2 동기화 흐름

```
1. 본 시스템 스케줄러 → 매일 06:00 트리거
2. 본 시스템 → 행정포털 조직도 API 호출
   GET {portal}/api/v1/departments?changed_since={last_synced_at}
   Header: Authorization: Bearer {service_account_token}
3. 행정포털 → 변경된 부서 목록 응답
4. 본 시스템 → 사용자 목록 API 호출
   GET {portal}/api/v1/users?changed_since={last_synced_at}
5. 행정포털 → 변경된 사용자 목록 응답
6. 본 시스템 → departments / users UPSERT
7. 본 시스템 → 부서장 변경 감지 시 target_assignees 자동 갱신 (트리거)
8. 본 시스템 → sync_logs INSERT (결과 기록)
```

### 2.3 부서 목록 응답 페이로드 (예상)

```json
{
  "total": 45,
  "departments": [
    {
      "external_id": "DEPT-0123",
      "name": "자치행정과",
      "parent_external_id": "DEPT-0100",
      "level": "과",
      "head_user_external_id": "hong.gd",
      "is_active": true,
      "updated_at": "2026-05-11T05:30:00+09:00"
    }
  ]
}
```

### 2.4 사용자 목록 응답 페이로드 (예상)

```json
{
  "total": 320,
  "users": [
    {
      "external_id": "kim.an",
      "name": "김안전",
      "email": "kim@damyang.go.kr",
      "phone": "063-xxx-xxxx",
      "department_external_id": "DEPT-0123",
      "position": "주무관",
      "position_code": "7급",
      "duty": "안전보건담당",
      "is_active": true,
      "updated_at": "2026-05-11T05:30:00+09:00"
    }
  ]
}
```

### 2.5 동기화 로직 상세

```
부서 UPSERT:
  FOR EACH dept IN 응답.departments:
    IF EXISTS (departments WHERE external_id = dept.external_id):
      UPDATE → dept_updated++
      IF head_user_id 변경 → head_changed++ + 트리거 발동
    ELSE:
      INSERT → dept_added++

사용자 UPSERT:
  FOR EACH user IN 응답.users:
    IF EXISTS (users WHERE auth_source='SSO' AND external_id = user.external_id):
      UPDATE → user_updated++
    ELSE:
      INSERT (source_type='PORTAL_SYNCED') → user_added++
      권한 자동 매핑 (permission_rules)

비활성 처리:
  외부에서 비활성된 사용자: users.is_active = false
  외부에서 삭제된 부서: departments.is_active = false (실제 DELETE 안 함)
```

### 2.6 오류 처리

| 상황 | 처리 |
|------|------|
| API 응답 없음 | sync_logs.result='FAILED' + 재시도 큐 + 운영자 알림 |
| 인증 실패 | sync_logs 기록 + 운영자 즉시 알림 (system 카테고리) |
| 데이터 정합성 오류 (예: parent_external_id 미존재) | 해당 행만 SKIP + 로그 기록 |

### 2.7 협의 필요 사항

- [ ] API 엔드포인트 URL
- [ ] 서비스 계정 토큰 발급
- [ ] changed_since 파라미터 지원 여부 (delta sync)
- [ ] 페이지네이션 정책 (대량 데이터)
- [ ] 페이로드 스펙 (부서·사용자)
- [ ] 동기화 주기 협의 (포털 측 부하)

## 3. IF-003 FMS 시설물관리시스템

### 3.1 개요

| 항목 | 내용 |
|------|------|
| 외부 시스템 | FMS 시설물관리시스템 |
| 연동 목적 | 시설형 관리대상(`target_type='FACILITY'`) 자동 동기화 |
| 방식 | REST API (Pull) |
| 주기 | 매일 06:00 + 수동 동기화 |
| 관련 SFR | SFR-002, SFR-016, SFR-020 |
| 본 시스템 화면 | TGT04-S FMS 동기화 설정 |

### 3.2 동기화 흐름

```
1. 본 시스템 스케줄러 → 매일 06:00 트리거
2. 본 시스템 → FMS 시설 목록 API 호출
   GET {fms}/api/v1/facilities?changed_since={last_synced_at}
3. FMS → 변경된 시설 목록 응답
4. 본 시스템 → targets UPSERT
   - 신규 시설: status='PENDING' (등록대기, 부서 지정 필요)
   - 변경 시설: cdpa_category, facility_grade, address 갱신
   - FMS에서 삭제: status='INACTIVE'
5. fms_sync_logs INSERT
6. 등록대기 1건 이상 시 운영자에게 새올 포틀릿 알림
```

### 3.3 응답 페이로드 (예상)

```json
{
  "total": 120,
  "facilities": [
    {
      "fms_id": "FMS-001234",
      "name": "군청 청사",
      "address": "전남 담양군 담양읍 추성로 1",
      "cdpa_category": "공중이용시설",     // 한글 → ENUM 매핑 필요
      "facility_grade": "1종",
      "owner_department": "자치행정과",     // 매칭은 본 시스템에서 (보조 정보)
      "is_active": true,
      "updated_at": "2026-05-11T05:30:00+09:00"
    }
  ]
}
```

### 3.4 cdpa_category / facility_grade 매핑

```
FMS 한글 값 → 본 시스템 ENUM:
  "공중이용시설"       → 'PUBLIC_USE'
  "원료·제조물"        → 'RAW_MATERIAL'
  "공중교통수단"       → 'PUBLIC_TRANSPORT'
  "1종" → 'GRADE_1', "2종" → 'GRADE_2', "3종" → 'GRADE_3'

매핑 미존재 시: status='PENDING' + 운영자 알림
```

### 3.5 신규 시설 등록대기(PENDING) 처리 흐름

```
1. FMS 동기화로 신규 시설 INSERT (status='PENDING', department_id=NULL)
2. 운영자(SHM+)가 TGT04-S에서 등록대기 항목 확인
3. [지정] 버튼 클릭 → 부서 선택 모달
4. department_id 설정 + status='ACTIVE' 전환
5. target_assignees RESPONSIBLE(AUTO) 자동 매핑 트리거 발동
```

### 3.6 협의 필요 사항

- [ ] API 엔드포인트 URL
- [ ] 인증 방식 (API Key / OAuth2)
- [ ] 페이로드 스펙 (한글 → ENUM 매핑 규칙)
- [ ] FMS 외부 URL 패턴 ([FMS에서 보기] 링크용)
- [ ] 동기화 주기 협의

## 4. IF-004 차세대 e호조

### 4.1 개요

| 항목 | 내용 |
|------|------|
| 외부 시스템 | 차세대 e호조 (회계시스템) |
| 연동 목적 | 도급계약 자동 import + 위험성평가 자동 생성 |
| 방식 | REST API (Pull) |
| 주기 | 매일 06:00 + 수동 동기화 |
| 관련 SFR | SFR-007, SFR-010, SFR-013 |
| 본 시스템 화면 | CON 모듈 / RSK04-S 설정 |

### 4.2 동기화 흐름

```
1. 본 시스템 스케줄러 → 매일 06:00 트리거
2. 본 시스템 → e호조 도급계약 API 호출
   GET {ehojo}/api/v1/contracts?changed_since={last_synced_at}
3. e호조 → 신규/변경 계약 응답
4. 본 시스템 → contracts UPSERT
   - 신규 계약: source='EHOJO_AUTO', status='DRAFT', step=1
   - target_id 매칭은 추가 로직 필요 (계약명·발주처 기반 추정 또는 운영자 확인)
5. 도급공사(CONSTRUCTION) 인 경우 risk_assessments 자동 생성
6. 알림 발송 (담당자 지정 필요)
7. sync_logs INSERT
```

### 4.3 응답 페이로드 (예상)

```json
{
  "total": 25,
  "contracts": [
    {
      "ehojo_contract_no": "2026-0123",
      "name": "청사 시설 보수공사",
      "contract_type_code": "01",              // 본 시스템 ENUM 매핑 필요
      "start_date": "2026-04-01",
      "end_date": "2026-08-31",
      "budget": 500000000,
      "funder_department": "자치행정과",
      "description": "...",
      "subcontractors": [
        {
          "company_name": "A건설",
          "business_number": "123-45-67890"
        }
      ],
      "updated_at": "2026-05-11T05:30:00+09:00"
    }
  ]
}
```

### 4.4 contract_type 코드 매핑

```
e호조 코드 → 본 시스템 ENUM:
  "01" 공사       → CONSTRUCTION
  "02" 용역       → SERVICE
  "03" 물품구매   → PURCHASE_INSTALL (설치 포함 시)
  "04" 위탁       → CONSIGNMENT
  그 외           → OTHER

매핑 규칙 협의 필요 (e호조 측 코드 체계 확인)
```

### 4.5 target_id 매칭 로직

```
e호조 계약 → 본 시스템 관리대상 매칭:
  Step 1. 발주처 부서명 → departments.name 매칭
  Step 2. 부서 산하 관리대상 중 후보 추출
  Step 3. 매칭 단일 후보 시 자동 연결
  Step 4. 매칭 다수 또는 없음 → 운영자 수동 매칭 필요 (PENDING)
```

### 4.6 도급공사 위험성평가 자동 생성 (SFR-007 7.1)

```
IF contracts.contract_type = 'CONSTRUCTION' AND 자동 생성 설정 ON:
  risk_assessments INSERT (
    target_id = contracts.target_id,
    name = contracts.name + ' 위험성평가',
    assessment_type = 'special',
    trigger_type = 'contract',
    source = 'EHOJO_AUTO',
    source_contract_id = contracts.id,
    cycle_stage = 'PREPARING'
  )
  notifications INSERT (대상자: 담당자, 카테고리: risk)
```

### 4.7 협의 필요 사항

- [ ] API 엔드포인트 URL

## 5. IF-005 새올 포틀릿 (알림 적재)

### 5.1 개요

| 항목 | 내용 |
|------|------|
| 외부 시스템 | 새올 시스템 (행정포털 내 알림 영역) |
| 연동 목적 | 시스템 알림을 새올 포틀릿에 적재하여 공무원에게 표시 |
| 방식 | REST API 호출 또는 DB 직접 적재 (협의) |
| 주기 | 이벤트 발생 시 즉시 (비동기 큐) |
| 관련 SFR | SFR-017 |
| 본 시스템 화면 | NTF01-L / NTF02-D / NTF03-S |

### 5.2 알림 적재 흐름

```
1. 본 시스템 이벤트 핸들러 → notifications INSERT (status='NEW')
2. 비동기 워커 → notifications 폴링
3. users.auth_source 확인:
   - SSO → 새올 포틀릿 API 호출
   - LOCAL → IF-006 SMTP 이메일 발송 큐 적재
4. 새올 API 응답:
   - 성공 → notifications.status='SENT', sent_at=NOW()
   - 실패 → notifications.status='RETRY', retry_count++
5. 재시도 정책에 따라 백오프 후 재시도
```

### 5.3 새올 포틀릿 적재 페이로드 (예상)

#### 방식 A — API 호출 (REST)

```
POST {saeol}/api/v1/notifications
Header: Authorization: Bearer {service_account_token}
Body:
{
  "recipient_id": "hong.gd",
  "title": "[결재] 안전경영방침 v2.5 결재 요청",
  "body": "온나라에서 결재 진행하세요",
  "link_url": "https://damyang-safety/policies/...",
  "category": "approval",
  "timestamp": "2026-05-11T14:23:00+09:00",
  "source_system": "damyang-safety",
  "priority": "normal"
}

응답:
{
  "success": true,
  "notification_id": "saeol-uuid",
  "displayed_until": "2026-05-18T14:23:00+09:00"
}
```

#### 방식 B — DB 직접 적재

```
새올 측 알림 테이블에 INSERT 권한 부여
스키마 협의 필요 (컬럼 명·타입·필수 항목)
이 방식은 트랜잭션 격리 등 운영 위험 있어 API 권장
```

### 5.4 카테고리 코드 매핑

| 본 시스템 카테고리 | 새올 측 분류 (협의 후 확정) |
|----------------|---------------------|
| approval | 결재 안내 |
| assignment | 지정·매핑 |
| compliance | 이행 임박·기한 |
| inspection | 점검 임박·결과 |
| risk | 위험성평가 |
| opinion | 의견청취 |
| contract | 도급계약 |
| system | 시스템 공지 |

### 5.5 사용자 매칭

```
본 시스템 users.external_id → 새올 공무원 ID 그대로 사용 (협의)
매칭 실패 시: notifications.status='FAILED' + error_message='recipient_not_found'
```

### 5.6 협의 필요 사항

- [ ] 연계 방식 (API vs DB 직접 적재)
- [ ] 새올 API 엔드포인트
- [ ] 인증 방식 (서비스 계정 토큰)
- [ ] 페이로드 스펙
- [ ] 카테고리 코드 매핑
- [ ] 표시 정책 (우선순위·유효기간·UI 디자인 가이드)

## 6. IF-006 SMTP 서버 (외부 사용자 이메일)

### 6.1 개요

| 항목 | 내용 |
|------|------|
| 외부 시스템 | SMTP 서버 (자체 또는 외부 메일 서비스) |
| 연동 목적 | 외부 사용자(SUB/CON) 알림 발송 (새올은 공무원 대상이므로) |
| 방식 | SMTP 표준 프로토콜 |
| 주기 | 이벤트 발생 시 (비동기 큐) |
| 관련 SFR | SFR-015, SFR-017 |

### 6.2 발송 흐름

```
1. 본 시스템 이벤트 핸들러 → email_queue INSERT (status='NEW')
   템플릿 코드 + 변수 데이터
2. 비동기 워커 → email_queue 폴링
3. 템플릿 렌더링 (제목·본문 HTML 생성)
4. SMTP 서버 연결 → 이메일 발송
5. 발송 결과:
   - 성공 → status='SENT', sent_at=NOW()
   - 실패 → status='RETRY', retry_count++ (최대 3회)
```

### 6.3 SMTP 설정 (협의)

| 항목 | 값 (예상) |
|------|---------|
| 서버 | smtp.damyang.go.kr (또는 외부 서비스) |
| 포트 | 587 (TLS) / 465 (SSL) |
| 인증 | SMTP AUTH (사용자명·비밀번호) |
| 발신 이메일 | noreply@damyang.go.kr |
| 발신자명 | "담양 안전관리시스템" |
| 보안 | TLS 1.2 이상 |

### 6.4 이메일 템플릿 목록

| 템플릿 코드 | 용도 | 변수 |
|----------|------|------|
| `invitation_email` | 외부 사용자 계정 초대 | 회사명, 발급자, 초대 링크, 만료일 |
| `password_reset` | 비밀번호 재설정 링크 | 이름, 재설정 링크, 만료 시간 |
| `contract_assigned` | 도급계약 배정 안내 | 회사명, 사업명, 발주처, 기간 |
| `consulting_assigned` | 컨설팅 위임 안내 | 회사명, 위임 부서·관리대상 |
| `contract_expiring` | 계약 종료 임박 (D-30) | 회사명, 사업명, 종료일 |

| 상황 | 알림 대상 | 카테고리 |
