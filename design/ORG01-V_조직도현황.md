## 4. 표시 항목

### 4.1 상단 영역

| 항목 | 데이터 소스 | 비고 |
|------|------------|------|
| 마지막 동기화 시각 | `MAX(sync_logs.finished_at) WHERE sync_type='PORTAL_ORG' AND result='SUCCESS'` | KST 표시 |
| `[수동 동기화]` 버튼 | — | CEO/GM만 활성 |

### 4.2 좌측 조직도 트리

| 항목 | 데이터 소스 | 비고 |
|------|------------|------|
| 부서 계층 | `departments` 테이블 재귀 (parent_id 기반) | is_active=true만 |
| 부서명 | `departments.name` | |
| 부서별 인원수 | `COUNT(users WHERE department_id=:dept_id AND is_active=true)` | |
| 부서장 표시 | `departments.head_user_id → users.name + position` | NULL이면 "미지정 ⚠" |
| 펼침/접힘 토글 | — | `▼`(펼침) / `▶`(접힘) |
| 군청 총인원 | `COUNT(users WHERE auth_source='SSO' AND is_active=true)` | 최상위 노드 |

### 4.3 우측 상세 패널 (부서 선택 시)

| 항목 | 데이터 소스 | 비고 |
|------|------------|------|
| 부서명 | `departments.name` | |
| 부서장 | `departments.head_user_id → users.name + position` | NULL이면 "미지정" |
| 상위 부서 | `departments.parent_id → departments.name` | |
| 인원수 (SSO / 직접등록) | `COUNT(users) GROUP BY source_type` | |
| 관리대상 수 | `COUNT(targets WHERE department_id=:dept_id)` | |
| 안전관리자 충족 | `dedicated_personnel` JOIN `target_assignees` 집계 | SFR-010 산출 |
| 보건관리자 충족 | 동일 | |

## 5. 데이터 스코프

| 권한 | 보이는 범위 |
|------|----------|
| CEO | 전체 군청 조직도 |
| GM | 본 부서 + 산하 (트리 root는 본 부서) |
| SM/SHM | 본 부서 + 본인이 담당하는 관리대상의 부서 |

| SUB/CON | 본 화면 접근 불가 (404 또는 권한 없음) |

## 7. 수동 동기화 동작

### 7.1 트리거

```
[수동 동기화] 클릭 → 확인 모달:
   "행정포털 조직도를 즉시 동기화하시겠습니까?
    예상 소요 시간: 약 N분"
   [취소] [동기화 시작]
```

### 7.2 진행 화면

```
[수동 동기화] 클릭 후 모달:
   ┌─────────────────────────────────┐
   │ 동기화 진행 중...                │
   │ ████████░░ 80% (12/15 부서)     │
   │ 부서: 12 처리됨                  │
   │ 사용자: 245 처리됨               │
   │                                  │
   │             [백그라운드 실행]    │
   └─────────────────────────────────┘
```

### 7.3 완료 처리

```
1. sync_logs INSERT 완료
2. 부서장 변경 감지 시 target_assignees(AUTO) 자동 갱신 트리거
3. 트리 새로고침 (마지막 동기화 시각 갱신)
4. 결과 토스트:
   "동기화 완료: 부서 +1, ~3 / 사용자 +5, ~10 / 부서장 변경 ~1건"
```

## 8. 자동 처리 로직

### 8.1 정기 자동 동기화

```
스케줄러: 매일 06:00
  → IF-002 행정포털 조직도 API 호출
  → departments / users UPSERT
  → 부서장 변경 감지 시 target_assignees(AUTO) 자동 갱신
  → sync_logs INSERT
```

### 8.2 동기화 실패 시

```
sync_logs.result = 'FAILED'
운영자(CEO/GM/SHM)에게 새올 포틀릿 알림 (category='system')
다음 정기 동기화 시 재시도
```

## 9. 오류 처리

| 케이스 | 메시지 | 처리 |
|--------|--------|------|
| 권한 부족 (SUB/CON 접근) | "접근 권한이 없습니다" | AUTH99-V `permission_denied` |
| 동기화 진행 중 [수동 동기화] 재클릭 | "동기화가 이미 진행 중입니다" | 토스트 |
| 동기화 API 오류 | "동기화 실패: {오류 메시지}. 운영자에게 알림 발송됨" | 토스트 + 자동 알림 |

## 11. 관련 DB 테이블

- `departments` (DB-001 §2.1): 부서 마스터 (재귀 조회)
- `users` (DB-001 §2.2): 부서별 인원수 집계
- `targets` (DB-001 §2.3): 부서별 관리대상 수
- `target_assignees` (DB-001 §2.4): 안전·보건 인력 충족 산출
- `dedicated_personnel` (DB-001 §6.1): 부서 단위 법정 인원
- `sync_logs` (DB-001 §3.1): 마지막 동기화 시각

## 12. 관련 인터페이스

- **IF-002 행정포털 조직도**: 정기·수동 동기화의 원천

## 13. 성능 고려

- 부서 트리 재귀 조회는 CTE 또는 materialized path 사용
- 인원수 집계는 캐싱 (5분) — 실시간 동기화 후 invalidate
- 안전·보건 충족 산출은 별도 view 또는 캐싱
